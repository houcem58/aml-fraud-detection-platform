# Frequently Asked Questions

## About the Project

**Q: Is this a production-deployed system?**

This is a production-ready platform with full engineering implementation. The architecture,
evaluation, and compliance framework are built to production standards. Deployment at a
live financial institution would require integration with the institution's core banking
system and regulatory certification specific to their jurisdiction.

**Q: What datasets were used for evaluation?**

The primary evaluation dataset is IBM AMLSim HI-Small (82,947 transactions, 6.24%
fraud rate), a synthetic AML benchmark generated by IBM's AMLSim simulator with
realistic transaction patterns and known fraud typologies. The IEEE-CIS Kaggle
Fraud Detection dataset was used for complementary tabular ML benchmarking.
Neither dataset contains real customer data.

**Q: Why is account-disjoint GroupKFold the honest evaluation protocol?**

Standard random K-fold leaks account-level features (`dest_risk`, `src_risk`) from
training to validation splits. In production, the model sees accounts it has never
scored before. GroupKFold (splitting by account) simulates this cold-start condition.
Under strict GroupKFold, node enrichment features collapse to ΔAUC ≈ 0.000 because
risk scores cannot propagate to unseen accounts. The research documents this gap
transparently — it is the central empirical finding of the manuscript.

**Q: Why does the GNN perform poorly under strict evaluation?**

HybridGAT achieves AUC 0.689 ± 0.048 under account-disjoint GroupKFold (vs 0.916
in warm-start). This is expected: GNN node embeddings rely on neighborhood aggregation,
which requires the account to appear in the training graph. Cold-start accounts have
no historical graph context, so the GNN defaults to uninformative embeddings.
XGBoost tabular stacking on 64-d GNN embeddings is more robust because the embedding
captures structural graph properties that generalize to unseen accounts.

## About the Technology

**Q: Why Kafka for AML fraud detection?**

AML detection must process transactions before they clear — typically a window of
seconds for instant payments and minutes for wire transfers. Kafka provides the
pub-sub backbone that decouples the payment API from the scoring engine, enabling
both horizontal scaling and reliable exactly-once processing semantics. The
`fraud.rlhf_feedback` topic also enables the feedback loop without coupling the
expert dashboard to the production scoring service.

**Q: Why Mistral 7B for audit reports rather than a larger model?**

Mistral 7B runs locally via Ollama with no external API dependency, keeping all
transaction data within the institution's perimeter. GDPR Article 5 and BCT data
residency requirements prohibit sending transaction data to external cloud APIs.
Mistral 7B is sufficient for verbalizing SHAP signals and graph pattern evidence
in compliance-grade language.

**Q: What is RLHF in this context?**

RLHF (Reinforcement Learning from Human Feedback) here refers to the expert
feedback loop: compliance officers tag reviewed transactions (confirmed fraud /
false positive / escalated), and this feedback is used to recalibrate model
thresholds and decision boundaries nightly. It is not RLHF in the full
policy-gradient sense used for LLM alignment — it is supervised recalibration
using expert labels as ground truth.
