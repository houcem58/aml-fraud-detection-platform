# Evaluation Results: AML Fraud Detection Platform

Evaluation code and trained model artifacts are in the private research repository.
Results below are from reproducible runs using IBM AMLSim HI-Small.

---

## Datasets

### IBM AMLSim HI-Small

The primary evaluation benchmark. Generated by IBM's AMLSim transaction simulator
with realistic AML typologies including layering, structuring, fan-out/fan-in, and
cycling patterns.

- **Transactions:** 82,947
- **Fraud rate:** 6.24% (5,176 fraudulent transactions)
- **Typologies:** layering, structuring, fan-out, fan-in, random (noise)
- **Split protocols:** P1 (strict GroupKFold), P2 (warm-start 80/20)

### IEEE-CIS Kaggle Fraud Detection

Complementary benchmark for tabular ML calibration.

- **Transactions:** 590,540
- **Fraud rate:** 3.5%
- **Purpose:** XGBoost feature engineering validation

## Evaluation Protocols

### P1 — Strict Account-Disjoint GroupKFold (primary)

Groups are defined by account ID. Each fold ensures that no account appears in
both training and validation sets. This simulates production cold-start: the model
must generalize to accounts it has never seen.

This is the **honest production-relevant protocol**. Node enrichment features
(`dest_risk`, `src_risk`) cannot propagate to unseen accounts, so their contribution
collapses under P1.

### P2 — Warm-Start (80/20 random split)

Standard random train/test split. Accounts appear in both training and test sets,
allowing enrichment features to carry historical risk information forward.
P2 scores are reported for comparison but should not be used to benchmark
production performance.

## Key Finding: Cold-Start Collapse

The central empirical finding of the research manuscript:

Under strict account-disjoint GroupKFold (P1), adding node enrichment features
(`dest_risk`, `src_risk`, computed from account historical risk scores) produces
**ΔAUC ≈ 0.000** — no measurable improvement.

**Root cause:** Enrichment scores are computed from an account's transaction
history. In P1, validation accounts are never seen during training, so their
enrichment scores are cold-start zeros — carrying no discriminative signal.
In P2, accounts appear in both splits, so their enrichment scores reflect
actual historical fraud patterns — explaining the large P2 performance gap.

This finding has direct practical implications: graph node enrichment should
not be the primary performance lever for systems that will encounter new
accounts at inference time. XGBoost tabular stacking on 64-d GNN embeddings
(which encode structural properties rather than historical labels) is more robust.

## Context-Aware Fusion Ablation

| Fusion Strategy | AUC (P2) | Delta vs Fixed Equal-Weight |
|---|---|---|
| Fixed equal weight (XGB=0.33, GNN=0.33, AML=0.33) | 0.901 | baseline |
| Fixed XGB-heavy (XGB=0.70, GNN=0.20, AML=0.10) | 0.931 | +0.030 |
| **Context-Aware Fusion (density-adaptive)** | **0.915** | **+0.014** |

Context-aware fusion underperforms fixed XGB-heavy weighting in P2 because
the warm-start GNN is strong on dense-network transactions. Under P1, the
context-aware approach is more stable as the GNN cold-start does not propagate
into isolated-transaction scoring.

## Reproducibility

All evaluation runs used:
- `random_state=42` for all splits and model initialization
- IBM AMLSim HI-Small v1.0 (public benchmark)
